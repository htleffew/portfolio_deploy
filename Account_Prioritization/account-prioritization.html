Dual-Model Priority Scoring.

Moving beyond single-metric classification by combining probability-to-convert with expected deal size to align machine learning outputs directly with business revenue goals.

IMPACT High-value account ranking
01 / The Single Metric Flaw

The Blind Spot in Conversion Probability

Let us look at a specific scenario: 6sense aims to enhance its account ranking system for sales outreach. Historically, machine learning systems in this space have relied on a single metric, which is the probability that an account will convert. You train a classification model, score your leads from 0 to 1, and hand the highest probabilities to the sales team.

This approach makes intuitive sense, but it has a significant blind spot. It treats all conversions as equal. A 90% chance of closing a $1,000 deal will always rank higher than a 10% chance of closing a $1,000,000 deal. To fix this, 6sense's business goal is to prioritize target accounts based on both their likelihood to convert and their expected deal size. By calculating a final prioritization score that balances these two factors, we can help sales teams allocate their time toward the highest-impact opportunities.

02 / Architecture

Data Architecture & Preprocessing

To explore this, we will use a simulated dataset of 6,000 records that mirrors real-world sales signals typically retrieved from a data warehouse like Hive.

Our feature space is divided into three main categories. We have firmographics (industry, region, company size, revenue), engagement signals (30-day page views, meetings booked, demo requests, email clicks), and sales metadata (sales owner ID, historical win status).

We are predicting two target variables:

  1. converted_next_90d: A binary target for our classification model. In our dataset, this has a roughly 9.47% conversion rate.
  2. deal_size_usd: A continuous target for our regression model. To mimic the strong positive skew of real-world sales deals (where most deals are average-sized, but a few are massive), we simulate this dummy data using a lognormal distribution (np.random.lognormal). However, the actual predictive model we build later will use a standard squared error objective.

Real-world data is rarely pristine, so we need a robust imputation strategy. For missing numerical features, we impute using the median. The median is highly resistant to outlier distortion, which is crucial for skewed data like revenue. For categorical features, we impute using the mode.

Next, we apply One-Hot Encoding to our categorical variables. This expands our feature space to 39 columns. We choose One-Hot Encoding over label encoding because it preserves category independence. It prevents the model from assuming that an artificial mathematical order exists between categories like "SaaS" and "Manufacturing," making the data highly compatible with tree-based models.

Before we train, we must address data leakage. Data leakage happens when you accidentally include a feature that acts as a proxy for the outcome. In our dataset, the historical_win feature is a prime suspect. Retaining features that incorporate future information or act as outcome proxies will artificially inflate your model's performance during training, but the model will fail entirely in real-world deployment. Following established data mining principles, we explicitly drop the historical_win column to maintain a strict separation between predictive features and target outcomes.

import pandas as pd
import numpy as np
from sklearn.model_selection import train_test_split
from sklearn.impute import SimpleImputer
from sklearn.preprocessing import OneHotEncoder
from sklearn.compose import ColumnTransformer
from sklearn.pipeline import Pipeline

# Drop leaky feature
df = df.drop(columns=['historical_win'])

# Define features and targets
X = df.drop(columns=['converted_next_90d', 'deal_size_usd'])
y_class = df['converted_next_90d']
y_reg = df['deal_size_usd']

# Preprocessing Pipeline
numeric_features = ['30d_page_views', 'demo_requests']
categorical_features = ['industry', 'revenue_tier']

numeric_transformer = SimpleImputer(strategy='median')
categorical_transformer = Pipeline(steps=[
    ('imputer', SimpleImputer(strategy='most_frequent')),
    ('onehot', OneHotEncoder(handle_unknown='ignore', sparse_output=False))
])

preprocessor = ColumnTransformer(
    transformers=[
        ('num', numeric_transformer, numeric_features),
        ('cat', categorical_transformer, categorical_features)
    ])

X_processed = preprocessor.fit_transform(X)
03 / Stage 1

Predicting Conversion Probability

Our core architecture relies on XGBoost for both predictive tasks. XGBoost is a highly scalable tree boosting system that handles sparse data beautifully and provides state-of-the-art performance for tabular datasets.

For the first stage, we use the XGBClassifier with a binary:logistic objective to predict the probability of conversion.

The main hurdle here is class imbalance. Only 9.47% of our accounts actually convert. If we do not correct for this, the model might simply learn to predict "no conversion" every time and still achieve high accuracy.

We handle this in two ways. First, we use a stratified 80/20 train/test split. Stratification ensures that both our training and testing sets maintain that exact proportion of positive cases. Second, we utilize the scale_pos_weight parameter in XGBoost. We set this to 9.56, which is the ratio of negative to positive instances. This tells the algorithm to heavily penalize misclassifications of the minority class, forcing it to pay attention to the rare conversion events.

from xgboost import XGBClassifier
from sklearn.metrics import precision_recall_curve, auc

# Stratified Split
X_train_c, X_test_c, y_train_c, y_test_c = train_test_split(
    X_processed, y_class, test_size=0.2, stratify=y_class, random_state=42
)

# Calculate scale_pos_weight (Negative instances / Positive instances)
scale_weight = (len(y_train_c) - sum(y_train_c)) / sum(y_train_c) # ~9.56

classifier = XGBClassifier(
    objective='binary:logistic',
    scale_pos_weight=scale_weight,
    random_state=42
)
classifier.fit(X_train_c, y_train_c)

When evaluating this model, standard accuracy is unhelpful. Instead, we track Precision, Recall, F1-Score, and ROC-AUC. Crucially, we prioritize the Precision-Recall Area Under Curve (PR-AUC) over ROC-AUC. Research shows that ROC plots can be visually deceptive in highly imbalanced scenarios. PR-AUC specifically evaluates the fraction of true positives among positive predictions, making it a vastly more informative metric for rare-event data.

04 / Stage 2

Predicting Expected Deal Size

For the second stage, we want to predict the monetary value of the deal. As noted earlier, while the underlying dummy data was simulated with a lognormal distribution to reflect real-world skew, we train the actual XGBRegressor using a standard squared error objective (objective='reg:squarederror').

There is a very specific training nuance here. We train this regressor exclusively on the subset of accounts that successfully converted (converted_next_90d == 1).

This is a mathematical necessity. Deal size is only relevant upon conversion. If we trained the regressor on non-converting accounts (which have a deal size of $0), the model would become confused, trying to learn the features of a $0 deal rather than learning what drives a large deal versus a small deal among actual buyers.

from xgboost import XGBRegressor

# Filter for ONLY converted accounts for regression training
converted_indices = y_train_c == 1
X_train_r = X_train_c[converted_indices]
y_train_r = y_reg.iloc[y_train_c.index][converted_indices]

regressor = XGBRegressor(
    objective='reg:squarederror',
    random_state=42
)
regressor.fit(X_train_r, y_train_r)

We evaluate this regression model using Mean Squared Error (MSE) and Root Mean Squared Error (RMSE) because the latter is highly interpretable (the error is expressed in actual US Dollars) alongside R-squared to measure the explained variance.

05 / Synthesis

Synthesizing the Priority Score

Now we bring the two models together. Our final ranking metric is calculated with a simple expected value formula:

Priority Score = P(convert) * E[deal_size]

You might notice an interesting dynamic here. We are applying our regression model (which was trained only on converters) to all accounts, including those that will likely never convert. Is this analytically sound?

Yes, it is. A low conversion probability naturally suppresses the final priority score. If the regressor looks at a tiny company and predicts a theoretical deal size of $100,000, but the classifier determines there is only a 0.1% chance of them buying, the final priority score becomes $100. The math elegantly self-corrects. As a final safety check, we clip all predicted deal sizes at a minimum of 0 to maintain ranking integrity.

Account P(convert) ▼ E[deal_size] ▼ Priority Score ▼ 
# Predict Probability of Conversion (P_convert)
# [:, 1] gets the probability of the positive class
p_convert = classifier.predict_proba(X_processed)[:, 1]

# Predict Expected Deal Size (E_deal_size)
e_deal_size = regressor.predict(X_processed)

# Clip negative predictions to 0
e_deal_size = np.clip(e_deal_size, a_min=0, a_max=None)

# Synthesize Priority Score
priority_scores = p_convert * e_deal_size

# Attach to original dataframe for ranking
df['priority_score'] = priority_scores
df_ranked = df.sort_values(by='priority_score', ascending=False)
06 / Interpretability

Interpretability with SHAP

If you hand a sales team a list of scores without explanation, they will likely ignore it. We need to provide transparency into the model's decision-making.

To do this, we utilize SHapley Additive exPlanations (SHAP). SHAP provides a unified, game-theoretic framework that assigns a unique importance value to each feature for a particular prediction. It allows us to show exactly why an account received a high score (for example, "This account scored highly because 30-day page views are elevated and they are in the SaaS industry"). It provides both local interpretability for individual predictions and global interpretability for the model as a whole, often visualized through summary beeswarm plots.

import shap

# Initialize JavaScript visualizations in notebook
shap.initjs()

# Explain the classifier's predictions using TreeExplainer
explainer = shap.TreeExplainer(classifier)
shap_values = explainer.shap_values(X_train_c)

# Generate a beeswarm summary plot
shap.summary_plot(shap_values, X_train_c, feature_names=preprocessor.get_feature_names_out())
07 / Deployment

MLOps and System Deployment

Transitioning this dual-model from a localized Python script to an enterprise Go-to-Market tool requires strict MLOps practices.

Models degrade over time as market conditions change. We monitor this concept drift using the Population Stability Index (PSI) and Kolmogorov-Smirnov (KS) tests. If we detect high drift, or if we see performance degradation (like drops in PR-AUC or R-squared), it triggers a retraining cycle. We can manage these workflows using orchestrators like Airflow or Kubeflow, often falling back on a fixed quarterly retraining schedule as a baseline.

For persistence and delivery, the models and preprocessing artifacts (like our median imputers and One-Hot encoders) are serialized using joblib, pickle, or MLflow. The scoring logic itself is deployed as a dynamic API using Flask or FastAPI. This allows the system to push scores directly into CRMs and visualization dashboards like Tableau or Power BI.

# Example FastAPI Deployment Snippet
from fastapi import FastAPI
from pydantic import BaseModel
import joblib
import numpy as np

app = FastAPI()

# Load serialized artifacts
classifier = joblib.load("xgb_classifier.pkl")
regressor = joblib.load("xgb_regressor.pkl")
preprocessor = joblib.load("preprocessor.pkl")

class AccountData(BaseModel):
    industry: str
    revenue_tier: str
    page_views_30d: int
    demo_requests: int

@app.post("/predict_priority")
def predict_priority(data: AccountData):
    # Convert input to format expected by preprocessor
    raw_data = pd.DataFrame([data.dict()])
    processed_data = preprocessor.transform(raw_data)
    
    p_convert = classifier.predict_proba(processed_data)[0][1]
    e_deal = max(0, regressor.predict(processed_data)[0])
    
    priority_score = p_convert * e_deal
    
    return {
        "probability_to_convert": float(p_convert),
        "expected_deal_size": float(e_deal),
        "priority_score": float(priority_score)
    }

Finally, we establish a qualitative feedback loop with the sales teams. Their on-the-ground experience is integrated back into the system to continuously refine the algorithm's utility.

By combining the probability of a win with the magnitude of the deal, 6sense moves away from single-metric optimization. The result is a highly logical, data-driven strategy that aligns machine learning outputs directly with business revenue goals.
